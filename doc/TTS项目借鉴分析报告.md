# TTS 开源项目借鉴分析报告

> 分析日期：2026-03-30
> 目标：为当前 MLX 本地 AI 助手项目（Mac M5 Pro + Windows RTX 3070 Ti）寻找 TTS 方案优化与功能扩展的借鉴点

---

## 一、当前项目架构概览

| 组件 | 技术栈 | 硬件 |
|------|--------|------|
| LLM 推理 | Qwen3.5-35B-A3B Q8 + mlx_lm | Mac M5 Pro (64GB) |
| TTS 服务 | Index-TTS + vLLM | Windows 11 / RTX 3070 Ti (8GB VRAM) |
| 客户端 | OpenClaw | Mac 局域网调用 |

## 二、四个项目综合分析

### 1. ebook2audiobookSTYLETTS2 / ebook2audiobook

| 属性 | 详情 |
|------|------|
| 地址 | https://github.com/DrewThomasson/ebook2audiobookSTYLETTS2 |
| 演进版本 | https://github.com/DrewThomasson/ebook2audiobook (⭐ 18.5k) |
| 核心能力 | 电子书 → 有声书，支持多 TTS 引擎、声音克隆、1158 种语言 |
| TTS 引擎 | XTTSv2、Bark、Fairseq、VITS、Tacotron2、Tortoise、GlowTTS、YourTTS |
| 输入格式 | epub、pdf、mobi、txt、html、rtf、doc/docx 等 20+ 种 |
| 输出格式 | m4b、mp3、flac、wav、aac、ogg 等（含章节元数据） |
| 硬件需求 | 最低 2GB RAM / 1GB VRAM，推荐 8GB RAM / 4GB VRAM |

**核心亮点：**
- **多引擎切换架构**：同一个框架支持 8 种 TTS 引擎热切换，低资源设备可选轻量引擎（YourTTS/Tacotron2），高资源设备用高质量引擎（XTTSv2/Bark）
- **SML 标签系统**：类 SSML 的标记语言，精细控制停顿、语速、角色切换
- **Calibre 集成的章节拆分**：自动检测和拆分电子书章节，每章独立生成音频后拼接
- **m4b 有声书标准输出**：输出带完整元数据（封面、章节标记、书名）的 m4b 文件
- **OCR 支持**：可处理扫描 PDF、图片中的文字
- **Apple Silicon (MPS) 支持**：明确支持 Mac 上的 MPS 加速

### 2. easyVoice

| 属性 | 详情 |
|------|------|
| 地址 | https://github.com/cosin2077/easyVoice |
| Stars | ⭐ 2,022 |
| 核心能力 | 超长文本 → 语音，AI 智能多角色配音，流式播放 |
| 技术栈 | Vue 3 + TypeScript + Element Plus（前端）；Node.js + Express（后端） |
| TTS 引擎 | Microsoft Edge TTS（免费）+ OpenAI 兼容 API |
| 部署方式 | Docker 一键部署 / Node.js 本地运行 |

**核心亮点：**
- **AI 智能配音推荐**：将文本分段后，由 LLM（OpenAI 兼容 API）分析每段内容，自动推荐最合适的声音、语速、音调、音量参数——这是非常值得借鉴的"LLM + TTS"联动思路
- **多角色 JSON 配音 API**：通过 `/api/v1/tts/generateJson` 接口，一次提交多个角色的台词和各自的配音参数，批量生成并自动拼接
- **流式传输**：无论文本多长，都可以立即开始播放，不需要等待全部生成完毕
- **免费无限制**：基于 Edge TTS，无时长和字数限制
- **完整的 Web UI**：开箱即用的前端界面，支持试听、预览
- **字幕自动生成**：语音合成同时输出字幕文件

### 3. Bark (Suno)

| 属性 | 详情 |
|------|------|
| 地址 | https://github.com/suno-ai/bark |
| Stars | ⭐ 39,063 |
| 核心能力 | 生成式文本→音频模型（不仅仅是语音，还含音乐、音效、笑声等） |
| 架构 | GPT 风格 Transformer（类 AudioLM / Vall-E） |
| 许可证 | MIT（可商用） |
| 支持语言 | 13 种（含中文简体） |
| 硬件需求 | 完整版 12GB VRAM，小模型版 2-8GB |

**核心亮点：**
- **非语言音频生成**：独特的情感标记系统——`[laughter]`、`[sighs]`、`[gasps]`、`[music]`、`♪` 等标签可直接嵌入文本
- **100+ 声音预设**：跨语言的丰富声音库，无需训练即可使用
- **自动语言检测与混合**：输入多语言文本时自动切换口音
- **HuggingFace Transformers 深度集成**：可直接通过 `AutoProcessor` + `BarkModel` 使用
- **小模型模式**：`SUNO_USE_SMALL_MODELS=True` 可大幅降低 VRAM 需求至 2GB 以下
- **CPU offload 能力**：`SUNO_OFFLOAD_CPU=True` 可将部分计算卸载到 CPU

**局限性：**
- 单次生成限制约 13-14 秒
- 不支持自定义声音克隆（仅预设声音）
- 生成质量有随机性，不是传统意义上的"精确"TTS
- 项目已较久未更新（2023年）

### 4. Qwen3-TTS（阿里通义千问）

| 属性 | 详情 |
|------|------|
| 地址 | https://github.com/QwenLM/Qwen3-TTS |
| Stars | ⭐ 10,094 |
| 核心能力 | 开源 TTS 系列，支持声音克隆、声音设计、流式生成 |
| 模型规模 | 0.6B / 1.7B 参数 |
| 支持语言 | 10 种（中、英、日、韩、德、法、俄、葡、西、意） |
| 架构 | 离散多码本 LM + Qwen3-TTS-Tokenizer-12Hz |
| 推理加速 | 支持 vLLM（day-0 支持） |
| 许可证 | 开源，支持 fine-tuning |

**核心亮点：**
- **三种生成模式一体化**：
  - **CustomVoice**：9 种预设高质量音色 + 自然语言指令控制情感/语速
  - **VoiceDesign**：通过自然语言描述创建全新声音（"17岁少年，男高音，说话时有点紧张"）
  - **VoiceClone**：仅需 3 秒参考音频即可克隆声音
- **Voice Design → Clone 工作流**：先用自然语言设计一个声音，再用 Clone 模式固化，实现跨多句一致的角色声音
- **极低延迟流式生成**：端到端合成延迟低至 97ms，输入第一个字就可输出音频
- **自然语言控制**：用 `instruct` 参数直接描述语气（"用特别愤怒的语气说"），无需调参
- **Qwen 生态统一**：与你目前使用的 Qwen3.5 LLM 同一生态，API 风格一致
- **可复用的 voice_clone_prompt**：创建一次声音特征，多次复用，避免重复计算
- **支持 fine-tuning**：可基于 Base 模型微调出自定义角色
- **vLLM 官方 day-0 支持**：可与你现有的 Index-TTS-vLLM 部署流程复用

## 四、可借鉴到当前项目的具体方案

### 方案 A：Qwen3-TTS 替换/升级 Index-TTS（强烈推荐 ⭐⭐⭐⭐⭐）

**理由**：与你当前的 Qwen3.5 LLM 同生态，且 TTS 质量在 Seed-TTS benchmark 上表现优异。

**实施路径**：
1. 在 Windows RTX 3070 Ti 上部署 `Qwen3-TTS-12Hz-0.6B-Base`（0.6B 模型显存足够）
2. 使用 vLLM-Omni 部署（与现有 Index-TTS-vLLM 流程类似）
3. 利用 VoiceClone 能力提供个性化音色
4. Mac OpenClaw 端通过同样的 HTTP API 调用

**可借鉴的能力**：
- **Voice Design → Clone 工作流**：让用户先用自然语言描述想要的声音，生成参考音频后固化为可复用的角色声音
- **3 秒声音克隆**：极低门槛的声音个性化
- **流式输出（97ms 延迟）**：OpenClaw 对话场景中实现"边生成边播放"
- **自然语言情感指令**：让 LLM 自动根据回复内容决定语气

### 方案 B：借鉴 easyVoice 的"LLM 驱动配音"架构（推荐 ⭐⭐⭐⭐）

**核心思路**：利用你已有的 Qwen3.5-35B LLM 能力，实现"AI 智能配音"。

**实施路径**：
1. LLM 生成回复文本后，额外生成配音控制参数（语气、语速、情感标签）
2. 将这些参数连同文本一起发送给 TTS 服务
3. TTS 根据参数生成更富情感表现力的语音

**示例工作流**：
```
用户提问 → Qwen3.5 生成文本回复 + 配音指令
         → TTS 服务根据指令合成语音
         → OpenClaw 播放
```

**可借鉴的具体设计**：
- easyVoice 的 `/api/v1/tts/generateJson` 多角色 JSON 接口设计
- 分段处理 + 流式拼接的长文本策略
- AI 自动推荐声音参数的 Prompt 设计

### 方案 C：借鉴 ebook2audiobook 的长文档转有声书能力（推荐 ⭐⭐⭐）

**适用场景**：如果你需要将文档/电子书转为有声内容。

**可借鉴的能力**：
- **Calibre 集成的章节智能拆分**：自动解析电子书结构，按章节独立处理
- **多引擎回退策略**：高质量引擎失败时自动降级到轻量引擎
- **m4b 标准有声书输出**：带封面、章节标记、元数据的专业有声书格式
- **SML 标签系统**：`<break time="1s"/>`、`<voice name="...">` 等精细控制标记
- **OCR 管线**：处理扫描文档或图片中的文字

### 方案 D：借鉴 Bark 的情感音效标签系统（选择性借鉴 ⭐⭐）

**适用场景**：增强 AI 助手的表达力。

**可借鉴的能力**：
- **非语言音频标签**：`[laughter]`、`[sighs]`、`...`（停顿）等
- **小模型 + CPU offload 策略**：在资源受限设备上的优化经验
- **声音预设库**：100+ 预设声音的组织方式

**注意**：Bark 本身已停更，不建议直接使用，但其情感标签的设计理念可以借鉴到 Qwen3-TTS 的 instruct 参数中。

## 六、技术风险与注意事项

1. **Qwen3-TTS 目前需要 CUDA GPU**：无法直接在 Mac M5 Pro 上运行，需继续使用 Windows 机器作为 TTS 服务端。未来如果 MLX 社区适配 Qwen3-TTS，则可全部迁移到 Mac 上
2. **RTX 3070 Ti 8GB VRAM 限制**：0.6B 模型应该可以运行，1.7B 模型可能需要 bfloat16 + 内存优化
3. **vLLM-Omni 目前仅支持离线推理**：在线 serving 还在开发中，短期内可能需要用 `qwen-tts` 原生包 + FastAPI 自行封装 API
4. **Edge TTS 的稳定性**：easyVoice 依赖微软 Edge TTS 免费 API，可能有并发限制和稳定性风险，不建议作为核心方案

---

## 七、结论

**最高价值的借鉴组合是：Qwen3-TTS（能力核心） + easyVoice 的 LLM 配音架构（交互设计）**

这个组合可以让你的 MLX 本地 AI 助手实现：
- Qwen3.5-35B 作为"大脑"生成文本 + 配音指令
- Qwen3-TTS 作为"嗓子"执行高质量语音合成
- 全 Qwen 生态统一，API 风格一致，长期维护有保障
- 流式对话体验：LLM 边生成文本 → TTS 边合成语音 → 用户边听

这将把你的项目从"文本对话助手"升级为"有情感表达能力的语音 AI 助手"。
